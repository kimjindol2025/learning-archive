## 🔍 세 가지 옵션의 정직한 분석

### Option A: 정공법 (FreeLang 언어 강화)

**작업**:
```
src/parser.ts의 primary() 함수에
ObjectLiteral 파싱 로직 추가

추가할 코드:
if (this.match('lbrace')) {
  const properties: Array<{ key: string; value: ASTNode }> = [];
  if (!this.check('rbrace')) {
    do {
      // "key": value 파싱
    } while (this.match('comma'));
  }
  this.consume('rbrace', 'Expected }');
  return { type: 'objectLiteral', properties };
}
```

**예상 기간**: 1주 (MEDIUM 난이도)

**상세 분석**:
```
부작업 1: Lexer (이미 준비됨)
  - lbrace 토큰 생성 ✅
  - 따옴표 처리 ✅
  - 시간: 0 (이미 완료)

부작업 2: Parser.primary() 수정
  - ObjectLiteral 파싱 추가
  - 예상 코드: 30-50줄
  - 시간: 2-3일

부작업 3: Evaluator 수정
  - ObjectLiteral 평가 추가
  - 예상 코드: 20-30줄
  - 시간: 2-3일

부작업 4: 테스트
  - ObjectLiteral 동작 검증
  - 예상 테스트: 10-15개
  - 시간: 1-2일

총 예상: 6-8일 (1주)
```

**장점**:
1. ✅ **원본 Lexer.fl 수정 불필요** - 완전히 그대로 실행 가능
2. ✅ **언어 완성도 향상** - FreeLang 자체가 강화됨
3. ✅ **Phase B-4 토대 마련** - Parser.fl, Checker.fl도 자동 지원
4. ✅ **재사용성** - 향후 모든 프로젝트에서 객체 사용 가능
5. ✅ **철학적 완성** - 렉서→파서→체커 완전 자체 호스팅 가능

**단점**:
1. ❌ **1주 대기** - 즉시 결과 못 봄
2. ❌ **구현 복잡도** - Evaluator까지 수정 필요
3. ❌ **테스트 필요** - 버그 가능성

**비용/편익**:
```
투자: 1주 (5일)
얻는 것: 영구적 기초 (무한)
ROI: 극도로 높음
```

### Option B: 우회법 (데이터 구조 변경)

**작업**:
```
Lexer.fl의 createToken()을 함수 호출 방식으로 변경

변경 전:
fn createToken(type, value, line, column) {
  return { "type": type, "value": value, ... }  ❌ 객체 리터럴
}

변경 후:
fn createToken(type, value, line, column) {
  arr = [type, value, line, column]
  return arr  ✅ 배열 사용
}

또는:
fn createTokenWithFields(type, value) {
  # ... 배열 또는 함수 기반 구조
}
```

**예상 기간**: 2-3일 (LOW 난이도)

**상세 분석**:
```
부작업 1: Lexer.fl 재설계
  - 객체 대신 배열/함수 기반
  - 예상 코드: 100-150줄 변경
  - 시간: 1-2일

부작업 2: 토큰 접근 방식 변경
  - token["type"] → token[0]
  - token["value"] → token[1]
  - 예상 변경: 50+ 줄
  - 시간: 1일

부작업 3: 테스트
  - Lexer.fl 동작 검증
  - 시간: 0.5일

총 예상: 2-3일
```

**장점**:
1. ✅ **빠름** - 2-3일로 즉시 결과 가능
2. ✅ **파서 수정 불필요** - 기존 코드 건드리지 않음
3. ✅ **로우 리스크** - 실패해도 기존 코드 안전
4. ✅ **검증 가능** - 렉서 로직이 작동하는지 확인 가능

**단점**:
1. ❌ **Lexer.fl 원본 설계 훼손** - 객체 구조 아님
2. ❌ **가독성 저하** - token[0], token[1] 등으로 불명확
3. ❌ **Phase B-4 불가능** - Parser.fl은 여전히 객체 필요
4. ❌ **임시방편** - 근본 문제 해결 안 함
5. ❌ **기술 부채** - 나중에 객체 추가 시 다시 수정

**비용/편익**:
```
투자: 2-3일
얻는 것: 임시 검증 (1-2주)
ROI: 중간 (단기) → 낮음 (장기)
```
